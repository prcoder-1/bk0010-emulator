# TODO: тайминг «медленной памяти» (видео-контеншен ЦП↔VP1-037)

**Статус:** отложено. Требует потактовой модели шины — крупная отдельная работа.
**Дата анализа:** 2026-07-15.

## Проблема

Тест таймингов **`BK_Tests/45com-lo.bin`** («CLOCK CYCLES METER 0.1 slow memory»,
©Manwe) на реальной БК-0010.01 показывает для 11 ячеек время на **+4 такта**
больше, чем выдаёт наш эмулятор. Наши базовые таблицы `Cpu::timingFor` верны
(проверены по XOP1/XOP2/XJMP/XBR/XSOB и совпадают с эталонным BKemu), поэтому
расхождение — именно эффект «медленной памяти», которого у нас нет.

Эталон (реальное железо) vs наш эмулятор — расходящиеся ячейки, все `+4`:

| Команда         | Наш | Реальная БК |
|-----------------|-----|-------------|
| `COM (R0)`      | 28  | 32 |
| `CLR (R1)`      | 28  | 32 |
| `CLR (R1)+`     | 28  | 32 |
| `CLR @(R1)+`    | 40  | 44 |
| `CLR @-(R2)`    | 44  | 48 |
| `DEC (R1)+`     | 28  | 32 |
| `INC (R1)+`     | 28  | 32 |
| `MOV (R0)+,(R1)+` | 40 | 44 |
| `MOV (R0),(R1)+`  | 40 | 44 |
| `MOV -(R1),@(R1)+`| 48 | 52 |
| `DEC @#fast`    | 40  | 44 |

## Почему это НЕ чинится простым правилом

Инструментированием эмулятора сняты точные адреса чтения/записи каждой измеряемой
команды. Результат доказывает, что штраф **не зависит ни от адреса, ни от опкода**:

```
CLR (R1)          пишет 020400            → реальная БК: 32 (МЕДЛЕННО)
MOV (R0)+,(R1)    пишет 020400 (тот же!)  → реальная БК: 40 (БЫСТРО)
MOV (R0)+,(R1)+   пишет 020400..025376    → реальная БК: 44 (МЕДЛЕННО, свип)
```

- `CLR (R1)` и `MOV …,(R1)` пишут в **один и тот же адрес 020400**, но времена
  разные → не адрес.
- `CLR` (RMW, чтение+запись = 2 обращения к 020400) — медленно; `MOV` (1 запись
  в 020400) — быстро; `MOV …,(R1)+` (свипящий указатель) — медленно.

Проверено эмпирически: любой статический адресный порог (`isSlow(addr)`) одни
ячейки чинит, другие ломает (порог 020000 дал +4 там где надо, но пере-штрафовал
`R0,-(R2)`, `MOV …,(R1)` и др.). Значит эффект — **потактовый контеншен шины**:
ждёт ли обращение ЦП, зависит от фазы видео-развёртки VP1-037 в этот такт. Цикл с
фиксированным адресом стабильно попадает в одну фазу; read-modify-write держит
шину 2 такта (чаще коллизии); свипящий указатель усредняется по фазам.

## Как это (не) сделано в референсных эмуляторах

- **`bk` (`/home/prcoder/emulators/БК-0010-01/bk`)** — `timing.c` только по опкодам,
  `access.c` без wait-state. Модели медленной памяти нет вообще.
- **BKemu / GID (`/home/prcoder/emulators/БК-0010-01/GID-src/BKemu`)** — модель:
  1. базовые таблицы по опкоду+режимам (совпадают с нашими);
  2. поправка **на каждое обращение к памяти**: `GetWordT/SetWordT` добавляют
     `nTimingCorrection` региона, CPU в конце делает `m_nInternalTick -= Σ`.
     Константы (`Config.h`): динам. ОЗУ `0`, статич. ОЗУ `2`, ПЗУ/регистры `4`;
  3. видеоконтроллер VP1-037 эмулируется точно для вывода
     (`Make_One_Screen_Cycle` каждые `3МГц/750кГц=4` такта), но **не тормозит ЦП**.

  **Для БК-0010 всё ОЗУ получает поправку 0** (`Board.cpp:1537`) — статические
  `2`/`4` только на платах EXT32 и БК-0011М. Итог: **BKemu тоже не воспроизводит
  45com-lo** и считает всё ОЗУ БК-0010 одинаково быстрым.

- **bk2010 / Java (`~/emulators/БК-0010-01/BK2010/bk2010-0.7`, декомпиляция jar)** —
  цикл-ориентированный: CPU (`K1801VM1`) копит `spentCycles`, в `genEA` добавляет
  `eaMemCycles[режим] = {0,12,12,20,12,20,20,28}` — **не зависит от адреса**.
  Видео (`updateRaster`) на каждом обращении к памяти подтягивает позицию луча по
  `getCycles()` ЦП, но **тактов ожидания процессору не добавляет**. Есть даже
  почти-готовый цикл-точный контроллер `VP1_37_Runner` (с фронтами
  `negedge_orig/posedge_orig`) — но он **никем не используется (мёртвый код)**:
  автор начал делать точную модель развёртки и не подключил. Итог: **bk2010 тоже
  не воспроизводит 45com-lo**.

Вывод: **ни один из трёх аккуратных эмуляторов** (`bk`, BKemu/GID, bk2010) эту
тонкость не моделирует — включая цикл-точный bk2010, где под неё даже начали писать
`VP1_37_Runner`, но не довели. Это подтверждает: эффект — потактовый контеншен
шины, который все обходят стороной.

## Предлагаемая реализация (если делать)

**Потактовая модель контеншена ЦП↔VP1-037.** Эскиз:

1. Вести счётчик такта в кадре (у нас 60000 тактов/кадр, 50 Гц) и по нему —
   состояние видео-развёртки: активная зона строки vs служебные видеоциклы
   (`bHgate`) vs кадровый гашение (`bVgate`). Портировать расписание из
   `Make_One_Screen_Cycle` BKemu (алгоритм VP1-037).
2. В `Cpu::rword/rbyte/wword/wbyte` (или на уровне `Board`) при обращении к
   разделяемой с видео памяти проверять, занят ли контроллер шиной в этот такт;
   если да — добавлять wait-такты до освобождения.
3. Учитывать тип цикла: DATI (чтение), DATO (запись), DATIP+DATO (RMW держит шину
   дольше — это объясняет `CLR (R1)` медленно / `MOV …,(R1)` быстро на одном
   адресе).
4. Верифицировать по `45com-lo` (эталон — фото реального железа) и убедиться, что
   XOP*/XJMP/XBR/XSOB и `cpu_tests` (59/59) не сломались.

### Риски / объём
- Затрагивает тайминг **всех** обращений к памяти → риск сломать уже верный
  тайминг «быстрой» памяти. Нужна аккуратная верификация по всем тестам.
- Точное расписание VP1-037 и правило wait-state придётся выводить итеративно;
  окупаемость невысокая (очень нишевый тест, игры от этого почти не зависят).
- Оценка: крупная задача (не «быстрый фикс»).

## Меньшая, независимая победа (можно сделать отдельно)

Портировать каркас BKemu «базовая таблица − поправка на каждый доступ» ради
корректной скорости доступа к **ПЗУ/регистрам** (`−4`/доступ). Наш `timingFor`
сейчас опкод-онли и это не учитывает. Улучшает общую точность профайлера/таймера,
но **45com-lo этим не чинится**.
