# TODO: построчный (растровый) вывод VP1-037 — точная картинка

**Статус:** отложено. Реализуемо (умеренная задача), но **сейчас не требуется** —
ни одна из наших тестовых игр не использует растровые эффекты (см. замер ниже).
**Дата:** 2026-07-16.

## Проблема

Наш рендер — «весь кадр за раз»: `Screen::render(mem)` вызывается в конце кадра и
использует **одно** значение регистров (`scroll_` `0177664`, видеорежим, палитра)
на все 256 строк. Поэтому **изменения видеорегистров в середине кадра теряются**.

На реальной БК VP1-037 защёлкивает состояние регистров в момент прохода луча по
строке, что позволяет **растровые эффекты**:
- **сплит-скролл** (скролящееся поле + статичная панель — регистр `0177664`
  меняется в середине кадра, на строке раздела);
- смена видеорежима по ходу кадра.

Мы такие эффекты сейчас рендерим неверно (применяем финальное значение регистра ко
всему кадру).

## Замер: нужен ли эффект нашим играм (2026-07-16)

Инструментировали запись в `0177664` по кадрам (5 игр, по 500 кадров, с нажатием
клавиши):

| Игра | Записей всего | Кадров с >1 записью |
|------|---------------|---------------------|
| DIGGER_V0.8 | 1 | 0 |
| VALLEY | 1 | 0 |
| KLAD2 | 2 | 0 |
| BK3D10 | 1 | 0 |
| DIGTIT_V0.8 | 1 | 0 |

**Вывод:** ни одна не пишет регистр скролла динамически (1–2 раза за весь прогон =
разовая инициализация), **ноль кадров с несколькими записями** → сплит-скролла нет.
Многие игры БК скроллят перерисовкой, а не регистром, поэтому это ожидаемо. Значит
построчный рендер для нашего набора даёт **нулевой видимый выигрыш**.

**Когда вернуться:** если появится демо/игра с растровыми эффектами (сплит-скролл,
per-line смена режима). Тогда сначала повторить замер на ней и убедиться, что она
пишет `0177664` (или регистр режима) в середине кадра.

## Предлагаемая реализация (когда понадобится)

Построчный «догоняющий» рендер, синхронизированный с тактами ЦП (как `updateRaster`
в bk2010). Потактовая точность ЦП **не нужна** — достаточно позиции луча по нашему
уже существующему счётчику тактов (`Board::totalTicks_`).

1. **`Screen`**: вынести цикл по строкам в `renderLines(mem, fromLine, toLine)` —
   логика построчного рендера уже есть (`Screen::render`, строки 30–39), просто
   параметризовать диапазон.
2. **`Board`**: маппинг «такты от начала кадра → номер строки луча» (учесть
   верхнее/нижнее гашение). Расписание развёртки взять из `Make_One_Screen_Cycle`
   (BKemu) или `updateRaster`/`VP1_37_Runner` (bk2010).
3. **Хук на запись видеорегистров** (`ioWrite`, `REG_SCROLL` `0177664`, режим,
   палитра): перед применением нового значения «догнать» растр до текущей строки
   луча старыми значениями, затем применить новое. В конце кадра дорисовать остаток.

### Объём / риск
- ~100–200 строк, точечно. Основная работа — корректный маппинг «такт → строка».
- **Риск низкий:** для статичных экранов (и всех наших текущих игр) вывод
  **бит-в-бит тот же**, регрессий нет.
- Ограничение БК-0010-01: палитра фиксированная (per-scanline палитры как на
  БК-0011М нет) → основной выигрыш это **скролл-сплиты** и смена режима.

## Отличие от «медленной памяти»

Это **другая, более лёгкая** задача, чем `[[slow-memory-timing]]`
(`docs/slow-memory-timing.md`): здесь нужна лишь позиция луча по тактам (картинка),
а не потактовый контеншен шины ЦП↔видео (тайминг). Одно без другого делается.
