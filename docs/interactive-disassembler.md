# Интерактивный дизассемблер / аннотатор

Цель: как в Ghidra — на лету называть метки, процедуры, поля данных, оставлять
комментарии, размечать код/данные; аннотации живут рядом с образом и переживают
перезапуск. **Не** встраиванием Ghidra (см. «Почему не Ghidra»), а своим
лёгким слоем поверх уже готового: `bk::SymbolTable`, `bk::disasm` (уже подставляет
символы), `Trace` (живой граф вызовов), MCP (`bk_xrefs`, `bk_type`).

## Почему не Ghidra

- Огромное JVM-приложение (SLEIGH, декомпилятор) — встраивание непрактично.
- В поставке **нет процессора PDP-11/К1801ВМ1** (`Ghidra/Processors/` без PDP-11):
  «из коробки» БК не дизассемблирует. Нужна SLEIGH-спека ВМ1 (крупная работа), а
  наш `bk::disasm` ВМ1 уже разбирает корректно.
- Наше преимущество: **живая отладка** (что реально исполнялось/вызывалось) — у
  статической Ghidra этого нет.

## План по срезам

### Срез 1 (✅ готово): редактируемые символы + комментарии + персистентность
- Core: `SymbolTable::set/remove` (правка), `Board` — карта комментариев
  (`setComment/comment/comments`).
- Оверлей: курсор (выбранная строка дизасма) — ЛКМ выбирает, ПКМ ставит точку
  останова; рендер комментариев и метки; подсветка курсора.
- GUI: клавиши в отладчике — `N` назвать/переименовать символ на курсоре, `;`
  комментарий; стрелки — двигать курсор. Пустой ввод удаляет.
- Персистентность: `имя.bkdb` (JSON: symbols[], comments[]) рядом с `.bin` —
  автозагрузка при загрузке образа, автосохранение при выходе, пункты меню.

### Срез 2 (✅ готово): разметка код/данные + типы
- `Board`: `DataType {Byte,Word,String,Ptr}` + карта `data_` (`setData/dataAt/…`).
- Оверлей рисует размеченные диапазоны как данные (`.byte`/`.word`/`.ascii`/
  `.word+символ` для указателя) и пропускает их при дизассемблировании; курсор и
  moveCursor учитывают данные.
- Клавиши в отладчике: `B` байт, `W` слово, `S` строка (авто-длина до NUL),
  `P` указатель, `U` обратно в код. Данные сохраняются в `.bkdb` (поле `data`).
- TODO среза: массивы произвольной длины через диалог, структуры, КОИ-7-строки.

### Срез 3: процедуры + список функций + навигация
- Процедура = диапазон + имя (+опц. сигнатура). Список функций (кандидаты — из
  профайлера/call-графа). Goto по имени, follow по цели, история назад/вперёд.

### Срез 4: xref-панель
- «Кто ссылается на адрес» (`bk_xrefs` уже есть) по клику; переходы.

### Срез 5: автоанализ из живой трассы
- Засев кандидатов процедур из фактических вызовов (`Trace` знает call-граф/edges)
  и из статических JSR-целей — уникально благодаря runtime.

## Формат `.bkdb`

```json
{ "symbols":  [ {"a": 512, "n": "start"} ],
  "comments": [ {"a": 668, "c": "счётчик кадров"} ],
  "data":     [ {"a": 680, "t": 1, "l": 8} ] }
```
`a` — адрес (десятичный), `n` — имя, `c` — текст; `data`: `t` — тип
(0=байт,1=слово,2=строка,3=указатель), `l` — длина в байтах. В срезе 3+ добавится
`procs` (диапазоны+имена).
